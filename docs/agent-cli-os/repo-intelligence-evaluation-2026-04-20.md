# Repo Intelligence Evaluation

Date: 2026-04-20

## Question

Should Agent CLI OS add a repo-wide knowledge or indexing layer so agents can understand a repository faster, use fewer tokens, and query structure instead of repeatedly grepping raw files?

## Short Answer

Yes, but the core should be a tool and artifact layer, not a big prompt skill.

The right shape is:

1. A deterministic per-repo index.
2. A compact query surface over that index.
3. An optional MCP server for repeated agent queries.
4. A thin front-door skill only if we need visible routing in chat.

The wrong shape is:

1. Another long always-on skill prompt.
2. A flat "pack the whole repo into one prompt file" default.
3. A hook-heavy installer that mutates every repo by default.

## Why This Category Matters

Recent graph and repository-context work consistently points in the same direction: repository-aware agents perform better when they query a structured representation of the repo instead of repeatedly rediscovering structure from raw files.

- Repository Intelligence Graph reports a mean 12.2% accuracy improvement and 53.9% lower completion time when agents receive a deterministic repository graph instead of only raw repository context.
  Source: https://arxiv.org/abs/2601.10112
- RepoScope argues that repository-level tasks fail when retrieval ignores structural relationships, and proposes a structural-semantic graph plus structure-preserving prompt construction.
  Source: https://arxiv.org/abs/2507.14791
- GraphCoder reports better repository-level completion by retrieving from a code context graph instead of relying on sequence-only context.
  Source: https://arxiv.org/abs/2406.07003
- Codebase-Memory presents a persistent Tree-sitter-based knowledge graph served through MCP for code exploration.
  Source: https://arxiv.org/abs/2603.27277

This is also visible in practice across current tools:

- Graphify builds a persistent graph, report, and optional MCP layer, and explicitly claims token savings from querying the graph instead of rereading raw files.
  Source: https://github.com/safishamsi/graphify
- CodeGraphContext is built around "CLI for developers, MCP for agents" and treats the graph as a persistent system between the code and the tools.
  Source: https://codegraphcontext.github.io/
- RepoMapper exposes a repository map through both CLI and MCP, prioritizing important definitions and relationships.
  Source: https://github.com/pdavis68/RepoMapper

## What The Best Current Tools Actually Do

### 1. Graphify

Best for:

- multi-modal corpora
- persistent knowledge graphs
- human-facing reports and visual exploration
- optional agent query through MCP

Important strengths:

- deterministic AST pass for code plus semantic extraction for docs and media
- queryable `graph.json`
- human-readable `GRAPH_REPORT.md`
- optional MCP server
- optional Obsidian-friendly output

Important evidence:

- "71.5x fewer tokens per query vs reading the raw files"
- `graphify.serve` exposes graph queries through MCP
- supports Codex-specific install and hooks

Sources:

- https://github.com/safishamsi/graphify
- https://github.com/safishamsi/graphify/issues/425
- https://github.com/safishamsi/graphify/issues/265

Important risks:

- install and hook idempotency can get messy in repos that already have their own integration patterns
  Source: https://github.com/safishamsi/graphify/issues/401
- non-deterministic semantic outputs create merge-conflict risk if graph artifacts are committed and regenerated by multiple people
  Source: https://github.com/safishamsi/graphify/issues/369
- large graphs hit renderer limits and need hierarchical or aggregated views
  Source: https://github.com/safishamsi/graphify/issues/447

### 2. Repomix and Gitingest

Best for:

- one-shot repo packaging for an LLM
- lightweight export
- simple "give me the repo as one artifact" workflows

Important strengths:

- dead simple
- easy to automate
- useful when the target model or interface expects one packed file

Important limits:

- they are prompt-packers, not persistent repo-intelligence systems
- they do not solve repeated querying or structural navigation very well
- they are better for one-off export than for an always-available repo brain

Sources:

- https://github.com/yamadashy/repomix
- https://davidesantangelo.github.io/gitingest/

### 3. CodeGraphContext / CodeGraph / RepoMapper

Best for:

- code-first deterministic indexing
- AST and relationship-aware querying
- MCP-native retrieval
- lower operational complexity than multi-modal graph systems

Important strengths:

- persistent local index
- explicit CLI plus MCP split
- direct architectural queries like callers, callees, impact, and file structure
- better fit for "repo understanding" than generic prompt packing

Sources:

- https://codegraphcontext.github.io/
- https://github.com/CodeGraphContext/CodeGraphContext
- https://github.com/colbymchenry/codegraph
- https://github.com/pdavis68/RepoMapper

## What This Means For Agent CLI OS

Agent CLI OS already inventories tools, apps, skills, MCP servers, and capability health. That solves control-plane discovery.

It does not yet solve repository-content discovery at the same quality level.

So the missing thing is real, but it is not "another skill" in the ordinary sense.

The missing thing is a repo-intelligence layer:

- persistent per-repo artifact
- fast query interface
- optional MCP exposure
- small, compact summary that agents can consult before raw search

## Recommendation

### Recommendation 1: Treat this as a capability with a tool core, not as a pure skill

If this lands, the core should be commands like:

- `agentcli repo-intel build`
- `agentcli repo-intel query`
- `agentcli repo-intel status`
- `agentcli repo-intel serve`

The skill, if added later, should be thin:

- `$repo-intelligence` or `$repo-map`
- only routes the agent to the CLI and the compact summary
- does not contain the actual knowledge or logic

### Recommendation 2: Start deterministic and code-first

Phase 1 should cover:

- files
- symbols
- imports
- calls
- top modules
- entry points
- changed-file impact
- test reachability

This gives most of the value with lower drift and lower maintenance cost than a semantic multi-modal graph.

### Recommendation 3: Make the first artifact a compact summary plus queryable index

The first useful outputs should be:

- a small human-and-agent summary file
- a structured machine-readable index
- a query command that returns only the relevant slice

Good examples:

- Graphify's `GRAPH_REPORT.md`
- CodeGraphContext CLI and MCP split
- RepoMapper's importance-ranked repo map

### Recommendation 4: Keep Obsidian export optional, not core

Obsidian is useful as a human-facing visualization and long-lived note surface, but it should be an export target, not the runtime contract.

Agents should query JSON and MCP.
Humans can optionally open:

- HTML graph
- wiki-style pages
- Obsidian vault export

### Recommendation 5: Do not make it always-on across all repos by default

Graphify issue history is a useful warning:

- installers can become invasive
- hooks can clash with repo-local workflow
- non-deterministic graph regeneration creates churn

Agent CLI OS should prefer:

- explicit `build`
- explicit `update`
- explicit `serve`
- optional repo-local hook setup only if the user asks for it

## Best Product Shape For Agent CLI OS

### Best near-term shape

Add a repo-intelligence capability that behaves like this:

1. Detect whether a repo-intelligence artifact already exists.
2. If yes, show status and expose query commands.
3. If no, explain what it can build and what it will cost.
4. Keep the default output tiny.
5. Keep the raw index behind query commands.

### Best long-term shape

Layered context:

1. Workspace map across projects.
2. Per-repo compact summary.
3. Repo-intelligence index or graph.
4. Raw files only when needed.

This matches the hierarchy proposed in Graphify issue #425 and the general direction of layered repository graph work.

## What Not To Build

Do not build:

- a giant always-on prompt that enumerates repo structure
- a flat single-file repo dump as the main workflow
- mandatory hooks for every repo
- an LLM-only semantic graph as phase 1
- a system that commits volatile graph outputs by default without a deterministic story

## Decision

This should be pursued.

But the correct first implementation is:

- capability first
- tool and artifact first
- deterministic and code-first first
- thin skill only if needed for visible routing
- optional MCP and optional Obsidian export after the core is solid
